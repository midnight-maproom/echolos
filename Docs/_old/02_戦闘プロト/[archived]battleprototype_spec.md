## 1. コンセプト（プロト版）

### 1.1 プロトの一行ピッチ
**国盗り × オートバトル × ローグライト** の3ジャンル融合タイトル『Echoes of the Lost Kingdom』の、コア体験（配置の読み・多戦線采配・周回構造）を最小コストで実証する縦スライス。

### 1.2 プロトで検証する 3 仮説（H1/H2/H3）

| # | 仮説 |
|---|---|
| H1 | 配置（前衛/後衛・射程・横列かばう）で結果が「読める」オートバトルは観るだけでなく**面白い** |
| H2 | 限られた手駒を複数戦線に割り振る「**どこに主力を向けるか**」の判断が面白い |
| H3 | ランダム手駒でやりくりし、成長で逓増する脅威を凌ぐ周回が面白い（ローグライトとして成立するか） |

仮説の詳細・成功判定基準は [200_prototype_policy.md §1, §6](200_prototype_policy.md) を参照。

### 1.3 想定プレイ時間
1ラン約 30〜60 分（プレイヤー慣熟度による）。1ラン完走で本プロトの全機能を体験できる。フル版の「1ラン2-3時間×3-10周≒30時間」とは別軸。

## 2. 物語と世界観（プロト版）

> 詳細なキャラクター・3エンディング分岐・周回シナリオの本実装は [100_game_design.md §2](100_game_design.md) で扱う。
> プロトでは**ナレ＋スチル＋フェード**による「物語の片鱗提示」のみを行い、対話・分岐・本シナリオは試遊版送り。

### 2.1 物語の片鱗提示の方針
- セリフバルーン・立ち絵カットイン・キャラクター対話なし
- **ナレ + スチル + フェード**のみで「これは情景提示」と分かる演出
- 結果画面の隅に「※物語層は試遊版フェーズで本実装」の小さい注記

### 2.2 タイトル＋導入チュートリアル
- キービジュアル（銀髪王女＋聖剣ロゴ＋過去の城と廃墟マップ・ガチ油彩）をフルスクリーン背景
- 「ゲームを始める →」主ボタンから導入チュートリアル → R1 開始
- 導入チュートリアルでは **背景説明スチル 1〜2枚＋短文ナレ**（Skip 可）

### 2.3 敗北エンド（「ありがとう」）
- 表示条件：いずれかの戦線が累計上限に達した／R7 ボス戦で勝てなかった
- 構成：銀髪王女涙のスチル → ナレ「王女は涙を浮かべて微笑んだ。『**ありがとう**』」→ 直接 R1 開始（タイトル飛ばし＝さっさと再挑戦）

### 2.4 勝利エンド（魔王の影・2段構成）
- 表示条件：R7 ボス戦に勝利（撃破 or 15ターン到達時の辛勝）
- **構成（2ページ「To be continued...」演出）**：
  1. **1ページ目（純ハッピー）**：青空＋抱きつき・魔王影なし → ナレ「—— 王国の危機は去った。」+「Thank you for playing!」→ 3秒表示後フェードアウト（「終わったか」と思わせる）
  2. **2ページ目（魔王示唆）**：同構図に魔王影＋紫雷雲 → ナレ最小「To be continued...」のみ → フェードアウト → タイトル画面へ
- **演出意図**：「Thank you for playing!」で完結を錯覚させてから To be continued で引き戻すコンソール RPG 風の伝統演出。完成版への引きを作る

### 2.5 R7 ボス劇場演出
- R6→R7 の遷移で「最終決戦」のヤマ場を視覚的に示す。「**R7　最終決戦**」大見出し＋ボス影絵スチル＋「街に何かが迫る——」ナレ → R7 内政フェーズへ
- 偵察後の配置/観戦フェーズではアイコンが正体（男爵 or サムライ）に差し替わる

実装の詳細・コミット履歴・素材は [220_prototype_roadmap.md §5.9〜§5.11](220_prototype_roadmap.md) を参照。

## 3. プレイ体験（プロト版）

### 3.1 プロトの 2 階層構造
フル版は**メタ（拠点周回）→ ラン → ラウンド → 戦闘**の4階層だが、プロトは **ラン → ラウンド → 戦闘** の2階層のみ（メタ層なし）。1ラン完走で物語が一周する構造に簡略化している。

### 3.2 段階構成（A→B のインクリメンタル）
プロト全体は4段階で構成される。段階1〜3 は仕様（本書）に対応し、段階4 は実装ロードマップ（300）に対応する。

```
【段階1】戦闘スライス（H1 検証）── 本書 §4
【段階2】多戦線ミニループ（H2 検証）── 本書 §5
【段階3】ローグライト・ミニループ（H3 検証）── 本書 §6
【段階4】ピッチ用プロト ── 220_prototype_roadmap.md
```

各段階の詳細な戦略的位置づけは [200_prototype_policy.md §4](200_prototype_policy.md) を参照。

### 3.3 1ラン全体の流れ（R1〜R7）

```
[初回起動]
  タイトル → 導入チュートリアル → R1 開始

[R1〜R6 共通フロー]
  ラウンド開始：各戦線（3本）へ敵パターンを抽選・割り当て
    → 【ドラフト】3択から1体加入（15%でレア抽選／R6 のみレア確定保証）
    → 偵察コマンドで脅威を可視化（1コマンドで3戦線すべて開示）
    → 内政（行動力2を4コマンドから配分）
    → 師団を各戦線へ配置
    → 全戦線を解決 → 戦線ごとの累計点数を更新 → 勝敗判定 → 次ラウンド

[R7 ボス戦の特記]
  R7 内政フェーズ開始時：R7 ボス劇場演出
    → ドラフトなし／内政あり（最後の調整機会）
    → 街固定でボス出現（平原/砦は敵なし＝全戦力を投入できる）
    → 撃破 or 15ターン辛勝 → 勝利エンド／敗北・引き分け → 敗北エンド

[エンディング]
  勝利 → エンディング2段構成 → タイトル
  敗北 → エンディング → 直接 R1（さっさと再挑戦）
```

### 3.4 1戦闘の流れ
1ラウンドで全戦線が解決される。1戦線の解決は複数ターンの戦闘から成り、戦闘ルールはフル版を流用（→ [110_combat_spec.md](110_combat_spec.md)）。

## 4. 戦術：配置と戦闘（プロト版）

> 戦闘ロジックの本体（ターゲット選択・かばう・ダメージ計算・状態異常・回避・蘇生など）は [110_combat_spec.md](110_combat_spec.md) を参照。
> ここではプロト固有の配置仕様・射程・戦闘エフェクトの土台を扱う。

### 4.1 画面と操作
- 自軍6体を **前衛3/後衛3** のグリッドに配置
- 敵はプリセット編成（数パターン）
- 配置確定 → オート進行（倍速・スキップ可）→ 結果表示
- 戦闘中、**リーダースキル**を手動発動（ゲージ消費）

### 4.2 射程（リーチ）3値
攻撃対象は「射程内の敵」から `予定ダメージ÷現在HP` 最大を選ぶ。届く相手の判定は [110_combat_spec.md §3](110_combat_spec.md) の**列単位の保護の原則**に従う。近接と中射程の差は「**後列から行動できるか**」の一点。

- **近接**：前列にいる時のみ行動可。届くのは「むき出しの敵」（=敵前列＋同列前列が死亡/不在のむき出し後列）。**後列にいると待機（置物）**
- **中射程**：位置を問わず行動可。届くのは「むき出しの敵」（同上）
- **遠隔**：位置を問わず敵前列・後列**どちらも**攻撃可。**列単位の保護を貫通**

※プロトで一時的に採用した「横列単位（前列が1体でも生きていれば後列全員に届かない）」の代替案は、固いタンク1人で後衛3人全員を守れてしまい配置の戦略性を消すため、**2026-05-30 に撤回**して列単位に戻した。

### 4.3 配置の仕組み（2×3 スロット）
- 射程下では「近接=前列／遠隔=後列」を自由に組めないと配置が機能しない（例：[近接1＋遠隔1] の2人師団＝近接を前列、遠隔を後列）
- **行（前列/後列）指定の配置＝射程の成立に必須**：割り当て順スロット割りを廃し、**前列/後列を選んで置ける**ように改修済み
- **列（3スロットのどれか）指定＝後回し**：扇形（隣接列）・貫通（後ろの列）といった**位置指定スキルが入って初めて意味を持つ**ため、プロト現状では列内の細かい位置選択は据え置く

### 4.4 戦闘エフェクト基盤（段階3 の土台）
段階3 の枠組みを成立させる戦闘要素群。詳細仕様は 110_combat_spec.md に準ずるが、プロトで実装している要素を以下に列挙する：

- **射程3値**（近接/中射程/遠隔）＋前列/後列の**横列指定配置**
- **横列かばう**（前列の盾は前列・後列の盾は後列を守る／タンクは攻撃不可）。戦闘開始時に自動付与
- **全体攻撃**（`AllEnemies`）、**必中**（`IsSureHit`）
- **攻撃技に状態異常/デバフ付与**（毒(燃焼)/麻痺/炎上/防御低下を攻撃に乗せる）
- **効果の時限化**（残ターン減算）＋**種類ごとのスタック上限**
- **防御無視フラグ**（技単位の割合無視＝サムライ等）
- **状態異常無効フラグ**（燃焼/毒/麻痺/凍結/呪いのみ無効。**能力デバフは別物**として弾かない）
- **麻痺・凍結中はかばう無効**（`IsCovering=false`）。▲プレイヤーの麻痺源は希少に
- **cleanse**（味方の状態異常治癒）／**dispel**（味方デバフ解除・軍師）
- **チャージ＝CD近似＋優先発動**：CooldownでNターン間隔を表現＋優先発動フラグで「解禁ターンに確定発動」。サムライ薙ぎ払い・全体魔法チャージ・アサシン開幕に共用
- **置物オーラ**（戦闘開始時付与＋死亡で剥奪）。パッシブスキルエンジンは不要（特殊配線）
- **`IgnoresCover` フラグ**（技単位のかばう貫通明示）。アサシンの開幕「暗殺」に使用
- **ロール × ロール特攻**（`Unit.Roles` × `MageHunterTag` 等）。攻撃側の内部処理タグと対象の公開ロールの組み合わせで、基礎ダメージに倍率を乗せるパッシブ機構

#### 基礎ダメージ算出の一元化（実装メモ）
評価フェーズ（`TargetEvaluator` のスコア計算）と実行フェーズ（`ActionExecutor` のダメージ計算）で「基礎予定ダメージ」を別経路で算出するとズレが生じる（実例：魔導士特攻倍率を実行側にだけ実装した結果、ターゲット選定が考慮せず雷魔導士ではなく司祭を狙ってしまうバグ）。これを避けるため、基礎ダメージは `BattleManager.ComputeBaseDamage(attacker, waza, target)` の単一経路で算出する。

- 内部処理：技固有のダメージ式 → 攻撃側パッシブ倍率（魔導士特攻など）
- 防御減算・最低1保証・回避判定は実行フェーズ側でのみ行う（評価フェーズは基礎値のみ参照）
- 将来の対象特攻（アンデッド特攻・機械特攻など）や属性相性倍率もこのヘルパーに集約する方針

#### Roles と Tags の使い分け（公開情報 vs 内部処理）
- **`Unit.Roles`**：ユーザーに公開可能な**ロール／種族カテゴリ**（`Mage`/`Archer`/`Healer`/`Tank` など）。UI のステータス画面・ドラフト時のフィルタ表示・ロール特攻判定の対象側で参照する。属性（`Element`：火/水/氷/雷/風/地/光/闇）とは独立した概念で、複数ロールを同時に持つことも可能
- **`Unit.Tags`**：戦闘ロジックの**内部処理用フラグバッグ**（`RowCover`/`AntiHealPassive`/`NoNormalAttack`/`MageHunter` など）。実装の起点として使う技術タグで、UI への露出は想定しない。パッシブの「**発動側**」フラグはここに置く（ロール特攻パッシブの場合：攻撃側=Tags、対象側=Roles）

#### 支援行動の選定（プロト調整・2026-06-04 変更）
仕様本体は [110_combat_spec.md §5](110_combat_spec.md) に新設。プロト固有の数値とこの変更の意図をここに記す。

- **回復発動閾値：HP90% 未満**（旧 HP50% 未満）
  - 動機：試遊で「ヒーラーがほとんど回復せず攻撃ばかりする」体験が悪く、Pitch 用にも見栄えが悪かった。「少しでも減ったら回復」を優先
  - 過剰回復の抑止は MaxHP クランプ＋実効回復 0 抑止に委ねる（110_combat_spec §5.3）
- **回復対象不在時のフォールバック：自己防御**（新規追加）
  - 行動者に DefenseUp（Magnitude=5・1ターン）を自己付与
  - 動機：後列の Healer1（遠隔）が「杖で前衛を殴る」絵的違和感の解消
  - 既存 StatusEffect.DefenseUp を流用するため軽量実装
  - 効果値 5 は控えめ（Paladin の専門バフより低い）。退化リスク（防御特化で壁になる）の抑制

### 4.5 リーダースキル
- プロトではリーダースキルは **1〜2種**で十分（例：全体回復／単体大ダメージ／1ターン防御強化）
- ゲージは戦闘経過で蓄積
- 王女＝「固定リーダー＋強スキル持ちユニット」として扱えば足り、物語要素は不要

## 5. 戦略：采配と領土（プロト版）

### 5.1 マップ（3戦線：平原・街・砦）
自**本拠地** ＋ 前線領地 3。敵が複数方向から攻めてくる構成。隣接する自領／敵領の境界に**戦線**が発生（同時に3戦線）。

| 戦線 | 点数上限 | 初期強化Lv | 最大Lv | 性格 |
|---|---|---|---|---|
| **平原（左）** | 6 | 0 | 1 | 余裕がある戦線。敵の侵攻を多少許容できるが、強化幅も狭い |
| **街（中央）** | 4 | 0 | 2 | 中庸。R7 で**ボスが固定で攻めてくる最終決戦の場** |
| **砦（右）** | 2 | 1 | 3 | 最も脆く、最も強化できる。事故率が高く全力守備が必要 |

#### 戦線ごとの性格設計
- **砦のピーキー設計**：累計2点（完敗1回 or 惜敗2回）で即GO。だからこそ偵察と拠点強化が要る。最大Lv3 まで強化できるので、しっかり投資すれば「攻防両方の重バフ」で難攻不落になる
- **街の中庸＋ボス決戦**：R1-R6 は普通の戦線として運用しつつ、R7 のボスを必ず受け止める。R7 までに街の累計点数をできるだけ温存しておく戦略意義がある
- **平原の余裕**：強化幅が狭く拠点強化Lv1 が天井だが、点数上限6なので捨て気味の運用も可能。「捨て戦線」候補

#### 戦線割り当てのルール
- R1-R6：弱／中／強パターンが各ラウンドの3戦線にランダム割り当て（重複なし＝同一ラウンド内で同じパターンが複数戦線に出ない）
- R7：街固定でボス出現。平原・砦には敵が来ない（最終決戦）

### 5.2 「どこに主力を向けるか」を成立させる仕掛け（核心）
プロトで H2 を最小コストで立証する非対称性：
- **手駒が全戦線をカバーできない**：強ユニットの数 < 戦線数。全戦線に主力は置けない
- **戦線ごとに敵強度が違う**：偵察しないと分からない。捨てる戦線／集中する戦線の判断が要る
- **捨てた戦線の帰結が結果に出る**：領地を失う／防衛ラインが下がる＝トレードオフが可視

### 5.3 1ラウンドの流れ（プロト簡略版）
```
内政（行動力を配分） → 師団を各戦線へ配置 → 全戦線が同時に戦闘解決 → 次ラウンド
```
- **行動力**：毎ラウンド一定回復（実装は2）
- 余った行動力は次ラウンドに繰り越せない

### 5.4 内政4コマンド
各ラウンドの内政フェーズ、プレイヤーは **行動力2** を4つのコマンドに配分する（繰越不可）。すべてのコマンドが行動力1を消費する。

| コマンド | 内容 |
|---|---|
| **偵察** | **3戦線すべて** に攻めてくる敵編成を **このラウンドだけ** 開示する。事前に脅威を読んで戦力配分と配置を決められる。1コマンドで全戦線分の情報を入手＝配分判断の核 |
| **兵種強化** | 兵種を選んで永続強化する（同一兵種は最大2回まで）。同名兵種を複数体保有していればすべてに適用される（§7.5） |
| **拠点強化** | 戦線を選んでその拠点強化レベルを1上げる。Lv上限は戦線ごと（§7.4） |
| **招集** | 3択ドラフトを行い、提示3体から1体を選んで加入させる（15%でレア抽選） |

### 5.5 同一アクション禁止ルール
**同一ラウンド内で、同じコマンドを2回まで実行することはできない**。行動力2を持つが、「招集を2連発」「砦を1ラウンドで2段階強化」「同一兵種を2回連続強化」などの単調な極振りは構造的に防がれる。

- 例：R3 で「兵種強化（双剣士）」と「拠点強化（砦）」は可。「兵種強化（双剣士）」と「兵種強化（騎士）」は不可（兵種強化アクション2回扱い）
- これにより、各ラウンドで **異なる軸の判断** を2つ強いることになる

### 5.6 治療コマンドはプロト版で廃止
本番（[100_game_design.md §5.5.4](100_game_design.md)）では「行動力とゴールドを消費してHPを回復する治療コマンド」を持つが、プロト版は資源系（ゴールド・建築資材）をオミットしているためHP回復を **ラウンド開始時の自動全回復** に簡略化した。これに伴い治療コマンドは内政から廃止し、4コマンド構成にしている。

### 5.7 建設・資材購入・イベントコマンドはプロト範囲外
本番（[100_game_design.md §5.5](100_game_design.md)）の内政コマンドは「建設／偵察／招集／治療／資材購入／イベント」の6種だが、プロトでは以下の3種をオミットしている：

- **建設**：地域への施設建設システム自体をオミット（建築資材を要するため）
- **資材購入**：建築資材をオミットしているため自動的に不要
- **イベント**：物語層を §2 の簡略提示に縮約しているため不要

プロト固有の **兵種強化** と **拠点強化** が成長軸として置換されている（§6.5）。

### 5.8 戦闘の報酬システムはプロト範囲外
本番（[100_game_design.md §5.4](100_game_design.md)）では戦闘解決後に「撃破スコア」で報酬テーブルが抽選され、資源（ゴールド・建築資材）・王国の記憶・ユニット3択ドラフト・装備などが報酬として獲得される。プロトでは：

- ユニット獲得は **ラウンド開始ドラフト＋招集** に集約（§6.4）
- 資源系・王国の記憶・装備はすべてオミット
- 戦闘結果は **戦線ごとの累計点数** に変換される（§6.6）

したがって戦闘の報酬システム（撃破スコア・報酬テーブル機構）はプロト範囲外。

## 6. 周回：ローグライト構造（プロト版）

### 6.1 検証仮説（H3 の意味）
**H3**：ランダムに与えられた手駒でやりくりし、成長（強化・入手）で逓増する脅威を凌ぐ周回ループは面白いか。
本作のコア欲求 ②**限られた選択肢でのやりくり感** ③**序盤の不完全さを面白さに変換** を直接検証する。

**成功判定（定性）**：「引きの中でうまく組めた」「次はこう回そう」「もう1周」。
理不尽な敗北（引き負けに祈るだけ）ではなく、**判断で挽回できた**と感じられること。
そして開発者自身が**試行錯誤（バランス調整）の段階で面白いと感じる**こと（＝この企画の継続可否の羅針盤）。

### 6.2 段階2 からの差分（段階3 の増築点）
| 増築する | 据え置き（段階2 のまま） |
|---|---|
| **戦線3つ（平原/街/砦）と戦線ごとの累計点数（負け回数）方式の勝敗**（旧：本拠地HP一本） | 1ラウンドの大枠（内政→配置→全戦線解決→次ラウンド） |
| 敵編成を毎ラウンド抽選＋強度逓増（R1-2弱／R3-4中／R5-6強／R7ボス） | 行動力（毎ラウンド2・繰越なし） |
| 味方を3択ドラフトで入手（毎ラウンド開始時＋内政「招集」） | 偵察（敵編成可視化） |
| 入手・**兵種強化**・**拠点強化**による成長（旧：攻撃強化/守備強化を置換） | 射程・横列かばう |
| **ラウンド開始時にユニットHP全回復**（プロト暫定・資源系オミットのため簡略化） | 負傷（HP0→翌ラウンド休み）・戦況評価 |
| **7ラウンド構成**（R7に街固定でボス出現＝最終決戦） | （旧6ラウンド構成を1延長） |

### 6.3 敵：ランダム襲撃＋エスカレーション
- 敵編成パターンを **弱／中／強の3層に分類** して複数定義（役割の偏りで配置の最適解が変わる：物量寄り／重装寄り／魔法寄り 等）。詳細は §7.7
- 毎ラウンド、各戦線（3本）に1パターンを **ランダム割り当て**（固定アンカー＋役割スコープ抽選）。R7 は街にボスが固定で出現
- **エスカレーションは2レイヤー**（味方が毎ラウンド増えるので、敵も数・質で追わせないと簡単になる）：
  1. **強度倍率**：難度ブロック（弱R1-R2／中R3-R4／強R5-R6）の1ラウンド目で ×1.00、2ラウンド目で ×1.05。R7 ボスは強の集大成として ×1.05。**難度ブロック境界でリセットしブロック内は小さく伸びる構造**。ブロック切替の段差は編成質の変化が担う＝二軸の二重課金を解消
  2. **ラウンドゲートのパターン解禁**：R1-2弱／R3-4中／R5-6強／R7ボスの教習ラダー（§7.6）
- 偵察で「今ラウンドその戦線に何が・どれだけ強く来たか」を可視化（情報で備えられる・このラウンドのみ）
- **R7 ボス出現の特記**：街に固定で出現するが、**どちらのボス（男爵 or サムライ）か** は偵察しないと判明しない。街への襲来そのものは事前通知される。平原と砦には敵が来ない＝最終決戦のため全戦力をボス戦に投入できる

### 6.4 味方：固定ピック＋3択ドラフト入手
- **開始3体は「姫騎士（固有キャラ）＋通常プール3択ドラフト2回」で構成**
  - 1体目＝固有キャラ **姫騎士**（前衛・固定加入・§7.2）。物語のあるキャラを最初から1体持たせて感情的なフックを作る
  - 2-3体目＝通常プールから3択ドラフト2回（プレイヤーの手で選ばせる）
- **毎ラウンド開始時に3択ドラフト**（R1-R6）：候補3体から1体加入。**15%でレアプール抽選**
- **R6 のレア確定保証**：同一ラン内で一度もレア抽選が走っていない場合、R6 のドラフトは確定でレアプール抽選になる
- **内政「招集」でも追加で3択ドラフト**（行動力1消費・同一ラウンド1回まで）。最大保有数はR1-R6で 3＋6＋（招集分）＝9〜15体程度
- 候補プール（通常12体／レア3体）は §7.3。**敵専用ユニット（散兵・ボス2体）と姫騎士は3択ドラフトに登場しない**
- **理想編成は組めない**＝今ある駒で戦線をやりくりする
- 純ランダム付与にしないのは、引き負けを**プレイヤーの選択で緩和**するため（本番でも必須の仕組み）

### 6.5 成長（兵種強化・拠点強化・入手の三本柱）
段階2の「攻撃強化／守備強化」（ユニット個体への永続バフ）は**廃止**し、以下に置換する。詳細数値は §7.4・§7.5。

- **兵種強化**（兵種単位の永続強化・行動力1消費）：兵種を選び、その兵種すべてのユニットに永続強化を適用する。**同一兵種は2回まで強化可能**。同名兵種をドラフトで複数体獲得していた場合、強化はすべてに適用される
- **拠点強化**（戦線単位の永続バフ・行動力1消費）：戦線を選び、その戦線の強化レベルを1上げる。戦線ごとに初期Lvと最大Lvが異なる（§5.1・§7.4）
- **入手による頭数増**：ドラフト・招集による戦線カバー力の成長
- **「同一ラウンド内で同じアクションは2回まで実行できない」制約**（§5.5）により、内政の単調な極振りは禁止される

#### HP回復はラウンド開始時に自動全回復（プロト暫定）
- ラウンド開始時、全ユニットのHPを最大値まで回復する（負傷状態のユニットも含めてHPだけは回復するが、負傷状態そのものは [110_combat_spec.md §1.4](110_combat_spec.md) / [100_game_design.md §4.4](100_game_design.md) 通り翌ラウンドの休養で解消）
- **これはプロト版限定の暫定簡略化**。本番版（[100_game_design.md §5.6](100_game_design.md)）では「戦闘ダメージは持ち越し」「ラウンド開始時に一定量自動回復」「HP回復には資源（ゴールド）が必要な治療コマンド」という構造になる予定

### 6.6 勝敗（戦線点数蓄積方式）
旧「本拠地HP一本」方式を **戦線ごとの累計負け点数** 方式に置き換える。3つの戦線にそれぞれ点数上限があり、いずれかが上限に達した時点でゲームオーバー。

#### 点数の付き方
| 戦線結果 | 累計点数 |
|---|---|
| 完勝・辛勝 | 0点（取られない） |
| 引き分け（ユニット減数同数） | 0点（取られない） |
| 惜敗 | +1点 |
| 完敗 | +2点 |

- **戦線ごとの勝敗判定**：戦闘開始時からの **ユニットの減った数の比較**。減数が少ない側が勝者、同数なら引き分け
- 勝利しても **過去に取られた点数は取り返せない**（あくまで蓄積方式）

#### クリア条件
- **R1〜R6 を3戦線とも累計上限未満で乗り切り、R7 のボス戦に勝つ**（撃破でなくともよく、15ターン到達時の辛勝で十分）

#### 敗北条件
- いずれかの戦線が累計上限に達する
- **R7 ボス戦で勝てない**（敗北・引き分けすべて即ゲームオーバー）

物語的エンド分岐は作らない（結果テキストのみ・実演出は §2 を参照）。

### 6.7 RNG の理不尽を避ける歯止め（設計必須）
- 味方入手は抽選ではなく **3択**（祈りゲー化の防止）。招集も同じ仕組み
- **R6 のレア確定保証**：同一ラン内で一度もレア抽選が走っていなければ R6 のドラフトは確定でレア。「レア未遭遇のまま終盤」を防ぐ
- **偵察**で脅威を可視化＝情報で備えられる（§5.4 偵察は1コマンドで全戦線開示＝配分判断の質を担保）
- 強度逓増は**決定的カーブ**（ランダムに跳ねない）＝負ける理由が読める
- ドロップ候補は役割が偏りすぎないよう抽選（必要なら同役割連続を緩和）
- **同一アクション禁止ルール**（§5.5）：行動力2でも単調な極振りは禁止。多様な選択を促す

## 7. コンテンツ（プロト版）

### 7.1 ユニット最小セット（射程との整合）
役割が割れて「配置の読み」が成立する最小構成。射程下では「近接は前列・遠隔は後列」が自然な形になる：

| 役割 | 配置 | 射程 | 特徴 |
|---|---|---|---|
| 重装前衛 | 前列 | 近接 | 高HP・高防で前列に居座り、**横列かばう**で同じ横列の味方を守る壁 |
| 前列近接アタッカー（剣士） | 前列 | 近接 | 高火力・低耐久。前列で晒される前衛アタッカー（守りが要る） |
| 機動アタッカー | 前列 | 近接 | 素早い前列近接。行動順で先制 |
| 魔法アタッカー | 後列 | 遠隔 | 防御無視 or **全体**。全体は同列かばうに集中させて盾を溶かす対策札 |
| 後列遠隔アタッカー（弓） | 後列 | 遠隔 | 高火力。前列の盾を無視して敵後列を狙う＝かばう対策 |
| 回復/バフデバフ役 | 後列 | 遠隔 | 味方HP回復・攻撃up/敵防御down。長期戦・編成の妙 |

> 「物理アタッカー＝後衛で守られる高火力」は射程下で矛盾する（近接が後列にいると殴れない）。**守られる高火力は遠隔（弓/魔法）**、**前列に出す高火力は近接（剣士）**として、前衛アタッカー/後衛アタッカーを射程で区別する。

### 7.3 ドラフト仕様
ドラフトは段階3 のやりくり感の核（§6.4・§6.7）。下記すべて 3択 から1体選んで加入する形式で統一する。

#### 適用される場面
- **ゲーム開始時**：固有キャラ「姫騎士」が1体固定加入 ＋ 通常プールから3択ドラフトを **2回** 実行（合計3体取得・2026-06-02 改訂）
- **R1〜R6 のラウンド開始時**：1回ドラフト（自動発生）
- **内政「招集」**（§5.4）：行動力1で1回追加ドラフト（同一ラウンド1回まで）

#### プールと抽選率
- **通常ドロップ**：レア以外の**12体**から3択（プールの構成は §7.2 の通常枠）
- **レアドロップ**：騎士・ニンジャ・軍師の**3体**から3択
- **レア抽選率**：各ドラフトで **15%** がレア抽選になる（残り85%は通常）
- **R6 のラウンド開始ドラフトのみ、同一ラン内で一度もレア抽選が走っていなければ確定でレアに固定**（レア未遭遇のまま終盤に入る事故を防止）
- **敵専用ユニット**（散兵などの雑魚ユニット・ボス2体）は**ドラフトに登場しない**

#### レア共存ルール
- 敵側でのレア3体（騎士・ニンジャ・軍師）の登場は **プレイヤーのレア確率と独立**（例：ボスのサムライ想定編成にニンジャは普通に出る）
- §6.7（理不尽回避）：レア3体はいずれも「唯一の答え」ではない専門家枠なので、ゲートしても祈りゲーにならない

### 7.4 拠点強化のレベル別効果
戦線ごとの拠点強化レベル（§5.1）が上がるごとに、その戦線で戦う味方ユニット全員に以下のバフが累積で適用される。

| Lv | 効果（累積） |
|---|---|
| Lv1 | その戦線で戦う味方全員に **PDEF +3 / MDEF +3** |
| Lv2 | Lv1 に加えて **ATK +5** |
| Lv3 | Lv1・Lv2 に **重ねて再度同じ強化**＝**PDEF +6 / MDEF +6 / ATK +10**（攻防両方の重バフ） |

> **設計変更履歴**：当初 Lv3 は「常時偵察済み」だったが、§5.4 偵察コマンドを「1コマンドで全戦線開示」に強化したことで自動偵察の意義が消えたため、攻防両方の重バフへ差替えた。

#### 数値設計の方針
- **定数加算**（パーセントではなく固定値）：低ステータスのユニットほど相対的に大きく恩恵を受ける
- **戦線×Lv 上限の振り分け**：平原は Lv1 まで／街は Lv2 まで／砦は Lv3 まで。最強の戦線である砦に最大の強化幅を割り当て、ピーキーな点数上限とトレードオフを成立させる
- 数値は実機検証で調整する

### 7.5 兵種強化のレベル別効果
兵種強化（§5.4）は **個体ごとに保持する強化レベル**（0〜2）として実装する。同一兵種でも個体ごとに異なる強化レベルを持てる構造とし、敵編成での「強化済みの双剣士＋未強化の双剣士」同居も成立させる。

#### 設計：強化レベルの管理
- **`Unit.EnhancementLevel: int`（0〜2）を新規追加**。各ユニットインスタンスが個別に強化レベルを保持する
- 戦闘ロジック側（`EffectiveATK` / 回避率 / `CalculateHealAmount` / `AppliedEffects.Magnitude` 等）が必要に応じてこの値を参照する
- **味方サイド**：CampaignState が「味方陣営の兵種ごとの現在の強化レベル」を保持し、ユニット加入時・兵種強化時にすべての該当ユニットへ自動適用
- **敵サイド**：編成定義時に個別ユニットの EnhancementLevel を指定可能

#### 兵種別の強化対象（1段階あたり）
「強みを伸ばす」原則で兵種ごとに強化対象を選定。回復役・バフ役・ニンジャは耐久ではなく **役割そのものを伸ばす特殊強化** で、「耐久上昇でカウンターが難しくなる」事態を避ける。

| 兵種 | 強化対象 | 1段階値 | 2段階上限 | 種別 |
|---|---|---|---|---|
| 一般タンク | PDEF | +3 | +6 | 基礎ステ |
| HPタンク | HP | +30 | +60 | 基礎ステ |
| 騎士★ | PDEF / MDEF | +2 / +2 | +4 / +4 | 基礎ステ（複合） |
| 双剣士 | ATK | +5 | +10 | 基礎ステ |
| サムライ | ATK | +5 | +10 | 基礎ステ |
| 大槌兵 | ATK / PDEF | +3 / +2 | +6 / +4 | 基礎ステ（複合） |
| アサシン★ | ATK | +5 | +10 | 基礎ステ |
| 弓兵 | ATK | +5 | +10 | 基礎ステ |
| ニンジャ★ | **回避率** | +5 | +10 | **特殊**（eva 15 → 20 → 25） |
| 炎魔導士 | ATK | +5 | +10 | 基礎ステ |
| 雷魔導士 | ATK | +5 | +10 | 基礎ステ |
| 司祭 | **回復量** | +10 | +20 | **特殊**（45 → 55 → 65） |
| 巫女 | **回復量** | +5 | +10 | **特殊**（14 → 19 → 24） |
| 踊り子 | **AttackUp Magnitude** | +3 | +6 | **特殊**（10 → 13 → 16・スタック上限3は据え置き） |
| 軍師★ | **置物オーラ Magnitude** | +2 | +4 | **特殊**（5 → 7 → 9） |
| **姫騎士（固有）**| **置物オーラ Magnitude** | **+2** | **+6（Lv3まで）** | **特殊／拠点Lv連動**（DefenseUp 5 → 7 → 9 → 11） |

#### 姫騎士の強化機構：拠点強化Lv連動（2026-06-02 確定）
姫騎士は兵種強化（内政コマンドによる強化）の対象外。代わりに **全戦線の拠点強化レベル合計** で動的に強化レベルが決まる。

- 算出式：`姫騎士 EnhancementLevel = 全戦線の拠点 BaseLevel 合計 ÷ 2`（切り捨て・上限 3）
- 反映タイミング：`Stage3RoundManager.StartRound` で `Stage3CampaignState.GetPrincessEnhancementLevel()` を計算し、姫騎士の `Unit.EnhancementLevel` に代入
- 既存の Magnitude 強化フロー（`BattleManager.InitializeBattle` の AuraEffect Magnitude 加算）にそのまま乗る

#### 拠点Lv合計と姫騎士強化Lvの対応表
| 拠点Lv合計 | 姫騎士強化Lv | DefenseUp Mag | 想定タイミング |
|---|---|---|---|
| 0-1 | 0 | 5 | R1-R2 |
| 2-3 | 1 | 7 | R3-R4 |
| 4-5 | 2 | 9 | R5-R6 |
| 6（最大：平1+街2+砦3） | 3 | 11 | R7 |

#### 副次効果：拠点強化の戦略意義が増す
- 既存：拠点強化Lv は戦線局所バフ（Lv1=DEF+3 / Lv2=ATK+5 追加 / Lv3=DEF+6 ATK+10）
- 追加：**拠点強化Lvが上がるほど姫騎士のオーラが伸びる** → 全戦線にまたがる固有メリット
- 「砦Lv3 まで上げる動機」が一段強くなる

#### 種別の説明
- **基礎ステ強化**：Unit のステータスフィールド（`MaxHP` / `BaseATK` / `PDEF` / `MDEF`）に EnhancementLevel × 値を加算する
- **特殊強化**：戦闘ロジックの該当箇所で EnhancementLevel を参照する追加分岐が要る：
  - 回避率：`Unit.BaseEvasion + EnhancementLevel × 5（ニンジャの場合）` 等を `TotalEvasion` で参照
  - 回復量：`CalculateHealAmount` デリゲートに強化分を加算
  - Magnitude：`AppliedEffects` のテンプレ生成時に強化分を加算（敵側で複数体同居する場合、各個体のテンプレが個別に強化される）
- **拠点Lv連動（姫騎士）**：兵種強化コマンドではなく、戦線の拠点強化レベル合計から動的に EnhancementLevel が決まる

### 7.6 敵パターン設計の原則
- **固定アンカー＋役割スコープ抽選**：パターンを「固定スロット＋プールから抽選するスロット」で定義（例：前列1=タンク固定／前列2-3=前衛プール抽選）。抽選は**ラウンド開始時**、偵察はその結果を開示（抽選後の振り直しはしない）
- 抽選プールは**役割でスコープ**（前列枠は前衛プール、等）。アーキタイプ（重装寄り/魔法寄り…）が崩れないように
- **教習ラダー（7ラウンド構成）**：
  - **弱(R1-R2)**＝対策なしで射程と基本配置を教える。雑魚ユニット主体（散兵など）の少人数編成
  - **中(R3-R4)**＝**タンクと後衛アタッカーが混じった編成**。前列タンクが立ちはだかり、後列の遠隔アタッカー（炎魔導士／弓兵等）が後ろから削る構造
  - **強(R5-R6)**＝敵も対策（全体/弓兵/麻痺等）を持ちプレイヤーの盲目的かばうを罰する。多役割の混成編成
  - **ボス(R7)**＝集大成。街固定でボス2種のいずれかが出現
- **編成サイズは6人前提ではない**：「限られた人員の中で、しっかり偵察して必要な人数だけ割り当てる」ゲーム性を重視する
- **直交の原則**：強度逓増（難度ブロック内 ×1.00 → ×1.05）＝「どれだけ強いか」、パターン抽選（弱→中→強の編成質UP）＝「どのユニットか」
- **敵専用プール**：散兵などの雑魚ユニットとボス2体は敵編成にのみ登場し、ドラフトには出ない

### 7.7 敵パターン具体定義（弱・中・強）
§7.6 の教習ラダーに従い、3戦線にユニークに割り当てる各3パターンを以下に定義する（重複なし＝同一ラウンド内で同じパターンが複数戦線に出ない）。R3-R4 や R5-R6 のような同一難度帯の連続ラウンドでも、ラウンドごとに再抽選される。敵編成の強化レベルは特に明示しない限り0（§7.5）。

#### 弱パターン (R1-R2)

| # | 編成 | 配置 | 想定難度 |
|---|---|---|---|
| ① | 敵なし | — | その戦線には攻めてこない。0体配置でOK |
| ② | 散兵×3 | 前列横並び | 2体で大体守れる、1体は厳しい |
| ③ | 散兵＋サムライ＋散兵 | 前列横並び（同順） | 2体以上必要。アタッカー（双剣士／サムライ等）を含まないとほぼ勝てない |

R1-R2 はプレイヤー手駒4-5体。①で「3戦線同時厳しい」を物理的に吸収する救済枠。

#### 中パターン (R3-R4)

| # | 編成 | 配置 | 想定難度 |
|---|---|---|---|
| ① | 散兵×2 | 前列横並び | 1体で対処可。弱パターン①「敵なし」を置換する**軽い救済枠**だが、完全放置は完敗（+2点）リスクが生じる点で弱①より厳しい |
| ② | 散兵×2＋重装兵＋炎魔導士 | 前列：散兵／散兵／重装兵、後列：炎魔導士（重装兵の真後ろ） | 3体配置を要求。重装兵が前列を保護、その間に炎魔導士が前列に火炎弾で削る |
| ③ | 散兵×2＋大盾兵＋弓兵 | 前列：散兵／散兵／大盾兵、後列：弓兵（大盾兵の真後ろ） | 3体配置を要求。HPタンクが時間を稼ぐ間、弓兵が必中遠隔で**味方の脆い後衛を後列に置いていると狙撃される** |

**②③の差別化＝偵察→配置工夫の余地**：敵後衛の射程が異なるため、偵察結果に応じて自軍配置を変える価値が出る：
- ②（炎魔導士＝前列のみ攻撃）：味方の脆い駒を**後列に置いてもOK**
- ③（弓兵＝後列含む必中遠隔）：味方の脆い駒は**前列に出すべき**

#### 強パターン (R5-R6)

| # | 編成 | 配置 | 想定難度 |
|---|---|---|---|
| ① | 双剣士＋弓兵 | 前列：双剣士、後列：弓兵 | **救済枠（だが中①より上）**。1体では対処不可、弓兵がいるため後衛に脆い駒を置けない |
| ② | 大盾兵＋双剣士＋司祭＋炎魔導士＋重装兵 | 前列：大盾兵／双剣士／（空）、後列：司祭／炎魔導士／重装兵（**後列タンク**） | 5-6体配置を要求。**後列タンク（重装兵）が後列をかばう＝弓兵が無効化される**。アサシンの開幕暗殺（IgnoresCover）で後衛除去が刺さる |
| ③ | 重装兵＋サムライ＋アサシン＋雷魔導士×2 | 前列：重装兵／サムライ／アサシン、後列：雷魔導士／雷魔導士／（空） | 4-5体配置を要求。**敵アサシン登場でプレイヤー後衛魔導士が即死＝魔導士運用不可** |

R5-R6 はプレイヤー手駒 8-9体（+招集分）。① 2体＋② 5-6体＋③ 4-5体＝計11-13体で**全戦線完勝は構造的に不可能**。

**「強ユニット無双を許さない」設計の表出**：
- 弓兵を主軸にしていれば②で機能不全（後列タンクが弓兵を無効化）
- 魔導士を強化しても③では出せない（敵アサシンで即死）
- いずれも偵察で事前判明するため、戦線割り当てを工夫すれば強化リソースは無駄にならない

## 8. 見せ方とプレイ時間（プロト版）

### 8.1 OnGUI ベースの UI 方針
- プロトは **OnGUI（IMGUI）で完結** する設計。Canvas/uGUI への本格移行は試遊版フェーズ（採択後）に回す
- 中央集約テーマ `Stage3GuiTheme` で色パレットと GUIStyle を一箇所管理
- アクセント色だけで状態を伝える寒色系ベースの配色

### 8.2 各フェーズのテンポ設計
- **観戦速度トグル**：コマ送り／×1／×2／×4
- **コマ送りモード**：自動 Tick を停止し、「次イベント →」ボタンで手動送り（スクショ撮影時の止め絵用）
- **スキップ**：この戦闘をスキップ／すべてスキップ
- **自動進行**：戦闘終了から 1.2 秒後に次戦線へ自動遷移

### 8.3 想定プレイ時間
- 1ラン約 30〜60 分（プレイヤー慣熟度による）
- 1ラン完走で本プロトの全機能を体験できる
- フル版の「1ラン2-3時間×3-10周≒30時間」とは別軸

### 8.4 動作環境
- **PC ブラウザ専用**（Chrome / Edge / Firefox / Safari いずれも可）
- **推奨解像度 1920×1080 以上**（Canvas 1920×1080 固定・2026-06-04 拡大）
- **入力**：マウス＋キーボード（ホバー操作を含む UI 設計）
- **スマホ・タブレットは動作対象外**（タッチでホバーが発生しないため一部機能が使えない・GUI スケーリングも未対応）

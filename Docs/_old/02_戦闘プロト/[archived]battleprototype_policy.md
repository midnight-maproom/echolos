# Echoes of the Lost Kingdom — プロトタイプ方針書

> **【アーカイブ通知】2026-06-09**
> 戦闘プロトの開発着手時に方針を定めたもの。本企画のコアのゲームシステム（オートバトル×SLG×ローグライト）のうち、オートバトル×SLGについて、ゲームとして成立するかを検証した。
> その後、本企画のUSPはゲームシステムの組み合わせそのものではなく、ゲームシステムとストーリーとのリンクであるという整理のもと、最小限のローグライト要素とストーリー体験をコアに据えた「VSプロト（Vertical Slice）」を制作するように方針転換。それに伴って戦闘プロトは開発停止した。
> 記録として残すが参照はしないこと。

> 本書は **戦闘プロトタイプ** の **方針** を定める。
> 「何を作るか／作らないか」の意思決定と、その**判断根拠**を整理する。
> 具体的な仕様は [210_prototype_spec.md](210_prototype_spec.md)、実装計画と進捗は [220_prototype_roadmap.md](220_prototype_roadmap.md) を参照。
> フル版仕様（本来やりたい全体像）は [100_game_design.md](100_game_design.md) / [110_combat_spec.md](110_combat_spec.md) を参照。

## 1. プロトの目的（検証仮説）

本プロトは**フル仕様の縮約版ではなく**、「3つの仮説を最小コストで検証する」ことだけに振り切った縦スライス。

| # | 仮説 | 段階 |
|---|---|---|
| **H1** | 配置（前衛/後衛・**射程**・横列かばう）で結果が「読める」オートバトルは、観るだけでなく**面白い** | 段階1 |
| **H2** | 限られた手駒を複数戦線に割り振る「**どこに主力を向けるか**」の判断が面白い（＝本作USP） | 段階2 |
| **H3** | ランダムに与えられた手駒でやりくりし、成長で逓増する脅威を凌ぐ周回が面白い（＝**ローグライトとして成立するか**） | 段階3 |

**証明しないこと（明確な非目標）:**
- 物語・キャラの魅力（立ち絵・CG・シナリオは作らない）
- メタ進行（王国の記憶・拠点強化）の中毒性
- コンテンツ物量（ユニット50-60種・八騎士・ボス階梯）
- 美術・サウンドのクオリティ

> プロトが証明するのは「**コアが面白いか**」だけ。魅力付けの要素は試遊版以降に回す。

## 2. スコープ（含む / 含まない）

| 含む | 含まない |
|---|---|
| ユニットの前衛/後衛配置・**射程（近接/中射程/遠隔）**・**横列かばう** | 物語・イベント・分岐エンド |
| ターゲット評価ロジック（既存コア流用） | 立ち絵・CG・キャラ作画 |
| リーダースキル介入＋ゲージ | メタ通貨・拠点復旧・恒久強化 |
| 倍速・スキップ・戦況評価（完勝〜完敗）・負傷制 | 八騎士・ボス階梯・聖剣 |
| 小マップ・複数戦線・行動力・最小内政 | 製品品質の美術・BGM（仮素材でよい） |
| ユニット 18 体（通常 15＋固有 1＋ボス 2） | ユニット物量・装備の作り込み |

詳細な含む／含まないの境界は [210_prototype_spec.md](210_prototype_spec.md) で章ごとに明示する。

## 3. オミットの理由（プロト範囲外とした判断根拠）

| 削ったもの | 理由 |
|---|---|
| 物語・立ち絵・CG | H1/H2/H3 の検証に不要。最も金と時間を食う領域 |
| メタ進行・拠点強化 | 「周回が楽しいか」は別の仮説。コアが面白い前提で後から乗る |
| ラン内資源（ゴールド・建築資材）・経済層 | コアの「配置の読み」「多戦線采配」とは独立。建設・治療・資材購入・戦闘の報酬テーブルなど経済層に紐づく要素全般を一括オミット |
| 八騎士・ボス階梯・聖剣 | コンテンツ物量。コアの面白さとは独立 |
| 固有ユニット加入システム・固有ユニット4名（仮）の物語 | 「結末を変えたい」USP軸の実装。コアの面白さ（H1/H2/H3）とは独立で、本番で協力者・編集人材と組んで設計。プロトは姫騎士（= 暫定エリシア）1名のみ |
| ユニット物量（60種） | 役割が割れる最小セットで「読み」は十分検証できる |
| 製品品質の美術・音 | 仮素材で代替。クオリティ証明は試遊版以降 |
| 装備（エンチャント） | 戦闘ロジックの本質ではない。ユニット側のステで読みは成立 |
| Canvas/uGUI 本格対応 | OnGUI で動くことを優先。本番で uGUI 移行（協力者起用前提） |

> このプロトの価値は「何を作ったか」ではなく「**何を作らずに核心だけ検証したか**」にある。

**追加検討事項として保留したアイデア**（例：クラスチェンジ／進化要素）は [400_production_tasks.md](400_production_tasks.md) の該当章に「💭 アイデア段階」として格納している。

## 4. 段階構成（A→B のインクリメンタル）

```
【段階1】戦闘スライス ── 単体で遊べるチェックポイント（H1を検証）
   既存の戦闘コアに最小UIを被せ、プリセット戦闘を配置→オート解決→介入できる状態
        │ 動いて「面白い」手応えが出たら拡張
        ▼
【段階2】多戦線ミニループ ── H2を検証
   小マップ＋2〜3戦線＋数ラウンド。行動力配分→師団配置→複数戦線が同時解決
        │ 「固定だと一度解けば終わるパズル」＝ローグライト未検証と判明
        ▼
【段階3】ローグライト・ミニループ ── H3を検証
   ランダム襲撃＋強度逓増／味方は3択ドラフトで入手／強化で成長／規定ラウンド防衛
        │ 「自分で1ラン完走＝コアはゲームとして成立」を確認
        ▼
【段階4】ピッチ用プロト ── 他者に伝えるための見せ物プロト（コンテスト応募物）
   段階3まで完成した縦スライスを「他人が触れる／動画で見せられる」水準に磨く。
   機能追加ではなく見せ方への投資（戦闘可視化／UIスタイル／チュートリアル）
```

**重要**: 段階1は「途中成果」ではなく**それ自体が完結して遊べる**こと。ここで H1 がダメなら段階2に進む前に立ち止まる。

> **戦闘土台の是正（段階3の前に挿入／2026-05-29 設計確定）**：配置・編成に意味を与える土台として、段階3に入る前に戦闘ルールを是正した。
> 順序は **① 射程（近接/中射程/遠隔＋前列/後列の横列指定配置） → ② 横列かばう＋対策（全体攻撃） → ③ 段階3** 。
> 理由：配置/編成が意味を持たないまま段階3のドラフト/逓増を積むと、検証したい「編成の妙味」がそもそも存在せず go/no-go 判断が空振りするため。

各段階の詳細な実装計画と進捗は [220_prototype_roadmap.md](220_prototype_roadmap.md) を参照。

## 5. 既存実装の流用と新規実装の方針

| 流用（実装済みCore） | 新規実装 |
|---|---|
| ターゲット評価・行動順・ダメージ計算 | 配置/戦闘のUI層 |
| 状態異常・多段・リアクション処理 | 戦況評価の表示・倍速/スキップUI |
| 戦闘単体のロジック | マップ・領地・戦線管理（段階2） |
| | 行動力・最小内政（段階2） |
| | 複数戦線の同時解決制御（固定順） |
| | ローグライト・ミニループ（段階3） |
| | ピッチ用見せ物プロト（段階4） |

> Core ロジックは流用できるため、プロトの新規作業は主に **UI層とマップ/戦線の薄い戦略層**。
> ここがフル版開発フェーズで挙げた「Unity 実装協力」の検証にもなる。

アーキテクチャ詳細は [500_architecture.md](500_architecture.md) を参照。

## 6. 成功判定（このプロトのゴール）

定量より**定性**で判断する。プレイテスト（自分＋数人）で：

- **H1**：「配置を考えた」「読んで勝てた／負けた理由が分かる」
- **H2**：「主力をどこに送るか迷った」「もう一回うまくやりたい」
- **H3**：「引きの中でうまく組めた」「次はこう回そう」「もう1周」
  - 理不尽な敗北（引き負けに祈るだけ）ではなく、**判断で挽回できた**と感じられること
  - 開発者自身が**試行錯誤（バランス調整）の段階で面白いと感じる**こと（＝この企画の継続可否の羅針盤）

この3つの手応えが出れば、**「重厚な戦略SLGとして多戦線の采配が成立し、配置が結果を決めるオートバトルとして読みが機能し、ローグライトとして周回できる」という USP（[100_game_design.md §1.4](100_game_design.md)）が実証された**ことになる。

段階4 完了時点では、**「動くもの＋それが面白い証拠」をコンテストで提示できる状態**を目標とする。

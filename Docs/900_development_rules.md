# Echolos 開発ルール

このファイルはプロジェクト全体で守る規約集です。CLAUDE.md からの参照で**毎回必読**。
個別の判断材料は `Docs/100_*` 以降の仕様書とメモリで補強しますが、ここに書かれた規約は最優先で守ること。

Claude 操作 Tips（並列 Edit 禁止・git mv・SO アセット生成手順等）は [920_implementation_notes.md §2](920_implementation_notes.md) を参照。本書は破ると設計が壊れる／作業事故が起きる／毎回の判断に効くルールに絞る。

## 1. Architecture Rules（絶対厳守）

### 1.1 4 層アーキ — Domain / Data / UseCase / Presentation
- バトルロジックやデータモデルは純粋な C# クラス（POCO）で構築し、絶対に `MonoBehaviour` を継承させない。
- 4 層の責務と依存方向：
  - **Domain**（`Assets/Scripts/Domain/`）：ドメインロジック・抽象 interface 定義。何にも依存しない。
  - **Data**（`Assets/Scripts/Data/`）：SO・データロード機構・Domain 抽象の実装。Domain にのみ依存。
  - **UseCase**（`Assets/Scripts/UseCase/`）：ゲーム進行ロジック。Domain 抽象にのみ依存（Data 実装を直接参照しない）。
  - **Presentation**（`Assets/Scripts/Presentation/`）：UI（MonoBehaviour）・Composition Root。全層依存可。
- 詳細は [500_architecture.md §1.1 / §2](500_architecture.md)。

### 1.2 状態の独立性
- 戦闘中の状態（バフ・デバフ・現在HP など）と、永続データ（最大HP・基礎ステータスなど）は完全に分離する。
- 永続データ（`Unit`）は戦闘中も変更されない原則。戦闘中の変化は `RuntimeUnit` 等のラッパーで管理する。
- 同原則は技にも適用：不変テンプレ `Waza`／戦闘中状態 `RuntimeWaza`（CD・使用回数）に分離。

### 1.3 イベント駆動 — 状態変化は API メソッド経由で必ず通知発火
- ロジック層が状態を変える際は **`public` フィールド直接公開を避ける**。
  - フィールドは `private` に
  - 読み取りは `IReadOnlyList<T>` 等の immutable view
  - 変更は `AddX` / `RemoveX` 等のメソッド経由のみ
  - メソッド内で `OnXAdded` / `OnXRemoved` 等のイベントを発火
- これは UI 連動忘れを**構造的に防ぐ**ための柱。プロト範囲でも必ず守る。本番（仕様 110）でも同方針。

### 1.4 asmdef は noEngineReferences=true（Domain / UseCase）
- `Echolos.Domain.asmdef` および `Echolos.UseCase.asmdef` は `noEngineReferences: true` 設定。
- **Domain / UseCase 配下では `UnityEngine.Mathf` 等の Unity API は使えない**。`System.Math` + 自前の `Clamp01` 等で代替する。
- これで Domain / UseCase 層を Unity から独立させ、Editor テストや本実装移植を容易にする。
- Data / Presentation は Unity API 使用可。

### 1.5 データ駆動原則 — 触りたいデータはコードに書かない
- バランス調整・シナリオライター作業・多言語化で触る可能性があるデータは、コードに直書きせず **SO + POCO の二段構成** で外部データとして持つ。コードは「構造」と「振る舞いの実装」のみを担う。
- 詳細・判定基準・SO 化候補一覧は [500_architecture.md §1.4 / §5.6](500_architecture.md)。

### 1.6 SO は 5 層パターン — Domain → Data 逆依存禁止
- SO 化対象は **POCO + SO ラッパー + Catalog 実装 + Domain 完成品 + Registry（必要なら）** の 5 層構成で実装する。
- **核心**：`I*Catalog` 抽象の戻り値に **Data 型（`*Definition` POCO）を入れてはいけない**。Catalog の `Get(id)` は Domain 完成品を返し、Data の POCO は Catalog 実装の内部だけが知る。逆依存になると 4 層アーキを破壊する。
- 詳細は [500_architecture.md §5.1 / §5.6](500_architecture.md)。

## 2. Platform Target

### 2.1 PC ブラウザ専用
- WebGL/itch.io の対象は **PC ブラウザのみ**（スマホ・タブレットは対象外・2026-06-03 確定）。
- ホバー前提 UI（ツールチップ等）を躊躇なく採用してよい。
- スマホ縦持ち・タッチ操作の考慮は不要。
- 基準解像度：Canvas 1920×1080。

## 3. Testing Strategy（テスト戦略）

### 3.1 テスト対象の限定
- テスト対象は `Assets/Scripts/Domain/` + `Assets/Scripts/UseCase/` 配下の POCO クラス（ドメイン／ユースケースロジック）のみ。
- `MonoBehaviour` や UI など、Unity の Play モードや画面描画に依存する部分の自動テストは書かない。

### 3.2 テストのタイミングと頻度
- 新しいコアロジック（ダメージ計算、ターゲット選定、状態異常の処理など）を実装する際は、必ずセットで NUnit（Unity Test Framework のエディターテスト）を用いた単体テストクラスを作成する。
- 実装とテストは「1つのタスク」として扱う。

### 3.3 エッジケースの網羅
- 仕様書にある複雑な条件（例：かばう時の多段攻撃処理、即死判定のシールド無視など）については、境界値やエッジケースを想定したテストメソッドを必ず含める。

### 3.4 契約変更時の既存テスト棚卸し
- イベント仕様・データ構造・API 契約を変更したら、**既存テストを必ず棚卸し**し、契約変更に追従させる。
- Grep で関連テストを洗い出してから本実装をコミットする。

### 3.5 テストファイルの配置
- テストコードは `Assets/Tests/Editor/Domain/` 配下に配置する（asmdef は `Echolos.Tests.Domain`）。
- 退避対象に依存するテストは `_old/` 配下に置き、`Echolos.Tests.Domain.Old` asmdef でコンパイル対象外化する（詳細は [500 §9.1](500_architecture.md)）。

### 3.6 テストメソッドの命名規則
- テストメソッド名に日本語を使用する場合、全角・半角を問わずカッコ `()` や `（）`、スペースなどの記号を含めない（コンパイルエラーの原因）。単語の区切りや補足にはアンダースコア `_` を使用する。

### 3.7 ヘッドレステスト不可・Test Runner はユーザー実行
- 本プロジェクトの開発環境ではヘッドレステスト実行不可（exit 198）。
- Claude 側でテスト実行確認はせず、ユーザーに Unity Editor の **Test Runner で All Tests 実行**を依頼する。
- 失敗があったら詳細（クラス名・行・期待値）をもらって修正する。

### 3.8 SO/Catalog テストの二系統使い分け
- **Domain / UseCase ロジックテストは Stub Catalog（インメモリ固定値）で決定論的に書く**。SO アセット非依存で CI 環境でも動く。
- **SO アセットの整合性（件数・主キー・パラメタ値・Registry 解決可能性）は別系統で本物の Catalog（`Resources.LoadAll`）を使って検証する**。SO アセット欠落時は fail する。
- Stub の固定値は本物 SO と同値に揃える（仕様準拠の数値を直書き）。整合性は本物 Catalog テスト側で担保。

## 4. Directory Structure

スクリプトを生成する際は、以下の上位構造を厳守する。詳細なサブ namespace・配置クラス一覧は [500_architecture.md §2.2](500_architecture.md) を Single Source of Truth とする（二重ドリフト防止・§8.4）。

| パス | asmdef | 役割 |
|---|---|---|
| `Assets/Scripts/Domain/` | `Echolos.Domain`（noEngineRefs=true） | ドメインロジック・抽象 interface |
| `Assets/Scripts/Data/` | `Echolos.Data` | SO ラッパー・Catalog 実装・SaveStore 実装（Editor ツールは `Data/Editor/`） |
| `Assets/Scripts/UseCase/` | `Echolos.UseCase`（noEngineRefs=true） | ゲーム進行ロジック・サービス |
| `Assets/Scripts/Presentation/` | `Echolos.Presentation` | MonoBehaviour・OnGUI・演出・Composition Root |
| `Assets/Resources/Data/` | （アセット） | SO アセット群 |
| `Assets/Resources/Icons/` | （アセット） | ちびキャラアイコン PNG（IconRegistry 経由） |
| `Assets/Resources/Images/` | （アセット） | スチル PNG・戦闘背景・マップ背景（BackgroundRegistry 経由） |
| `Assets/Resources/Fonts/` | （アセット） | フォント（GuiTheme.JapaneseFont 経由・WebGL 日本語表示用） |
| `Assets/Tests/Editor/Domain/` | `Echolos.Tests.Domain` | NUnit テスト |
| `Assets/Scenes/` | — | シーンファイル本体（命名規則は §4.1） |

### 4.1 シーン命名
- **本番系**: `EcholosProto_*.unity`（『Echoes of the Lost Kingdom』略称 echolos + プロト）。Build Settings 登録対象。
- **デバッグ系**: `Debug_*.unity` プレフィックス。Build 対象外。`Presentation/DevTools/` 配下のスクリプトとセットで使う。

### 4.2 陳腐化資産の退避（`_old/`）
- 段階移行で不要になったシーン・スクリプト・仕様書は**削除せず `_old/` 配下に退避**する。
- スクリプトの退避先には専用 asmdef を配置し、`defineConstraints: ["ECHOLOS_OLD_PROTO"]` ＋ `autoReferenced: false` でコンパイル対象外化する（詳細は [500 §9.1](500_architecture.md)）。
- 削除すると履歴を辿るしかないが、`_old/` に置くと「あった」事実が物理的に残る。

## 5. Development Process

### 5.1 仕様書整合性
- 新しい機能を実装する際は、既存の仕様書の内容と矛盾しないか必ず確認する。実装内容に応じて関連ドキュメントだけ参照する（一覧は [CLAUDE.md](../CLAUDE.md) を SSoT とする）。
- 矛盾がある場合は実装前にユーザーへ確認し、仕様書 or 実装どちらを正にするか決める。
- **910 番台は対象外**：`910_*` は Claude が経緯・作業ログを雑記する共有メモ帳。仕様書の一括修正・整合性チェック・参照リンクの対象外（[8.7](#87-910番台ドキュメントの位置付け) 参照）。

### 5.2 こまめなコミット
- ファイル変更の区切りでこまめに git commit する（事前承認済みルール）。
- 1コミット 1論点を原則：「リファクタコミット」と「機能追加コミット」を分けるなど。
- コミット前のユーザー確認は不要。コミット後にテスト実行を依頼する。

### 5.3 コミット時の仕様書・メモリ同期確認
- コミット作成時には毎回、関連する仕様書・メモリの更新要否を**先に確認**する。
- 必要なら同じコミットか直後のフォローコミットに含める。
- これを怠ると Compact 直前に複数の更新漏れが一気に発覚する。

### 5.4 コミットメッセージ
- 全文日本語。タイトル + 詳細（変更理由・影響範囲・関連メモリ）。
- 最後に `Co-Authored-By: Claude Opus 4.7 <noreply@anthropic.com>` を必ず付ける。
- HEREDOC で渡してフォーマットを保つ。

## 6. Game Design Review

### 6.1 退化戦略の点検
- 新しい仕掛け（戦線強化・防衛強化・ボス調整など）を導入したら、「**コアの判断を回避できる退化戦略**にならないか」を設計段階で点検する。
- 退化が見えたら、退化戦略のリターンを抑えるか、コア判断を強制するメカを足す。

### 6.2 耐久構成のトレードオフ
- 耐久構成は本質的に強い（時間を稼げる）。「完全に詰む」より「やや不利」で折り合うバランスを取る。
- vs 耐久編成の対抗手段（毒・割合ダメージ等）は必ず 1 つ以上用意する。

## 7. AI Assistant Behavior & Output Rules

### 7.1 実装前の方針報告（Propose Before Implement）
- 実装に着手する前に「現行実装の確認結果」と「実装方針」を報告し、ユーザー承認を受けてから着手する。
- 軽微修正（タイポ・1行修正・コメント追加）や事前承認済み（こまめなコミット等）は除外。
- 報告フォーマット：(1) 現行実装の状態 (2) 提案する変更 (3) 影響範囲 (4) 工数感（**ただし時間見積もりは書かない**・分単位の見積もりは外す）。

### 7.2 既存パターンを疑う姿勢
- 既存コードを使う前に「このコードは**何を保証しているか／何を保証していないか**」を明示的に問う。特に層間の責務分担。
- 「実装されている＝正しい」と疑わずに踏襲しない。過去の Claude が「とりあえず動く」状態で完了判定したものを、後続セッションで「完了済」として誤継承するパターンは特に注意。
- 動作確認は「想定シナリオで実際に表示されるか」までやる。コードを読んで「動いてそう」で完了しない。

### 7.3 設計レビュー時のエッジケース・章間相互作用の積極指摘
- 仕様書レビューや設計議論で、エッジケース・章間の相互作用・曖昧点を**積極的に指摘**する。
- 「無難に同意」より「気になる点を全部出す」方が価値が高い（事前承認済み）。
- 指摘した上で「ユーザーが判断する」分担。

### 7.4 Compact 後の推測は明示フラグ
- Compact で抜けた文脈を推測で埋めた場合、**確定扱いせず明示的にフラグ**する（例：「Compact 前の発言を推測で補っています：……」）。
- ユーザーが訂正の機会を持てるようにする。

### 7.5 Unity エディタ操作の指示
- コードの生成だけでなく、必要なセットアップ手順をステップバイステップで提示する（GameObject 作成・コンポーネントアタッチ・SerializeField への紐付け・Prefab 化等）。
- ユーザーは Unity エディタの基本操作にあまり慣れていない前提。専門用語には補足情報を加えるか、具体的なメニューパス（例：`GameObject > Create Empty`）を記載する。

### 7.6 コード出力とスタイル
- **Edit ツールで差分編集を優先**。新規ファイル作成時のみ Write を使う。
- ファイル名は必ず該当ファイルパスを明示する（Markdown リンク形式：`[file.cs](path/to/file.cs)`）。

### 7.7 ライブラリとパッケージの使用方針
- 原則として、Unity の標準機能（Built-in 機能）のみを使用して実装を進める。
- 外部ライブラリや追加パッケージを使用すると劇的に効率や品質が上がる場合は、実装前に必ず提案し、導入のメリットと手順を説明して許可を得る。
- 無断で外部依存関係を増やさない。

### 7.8 SO アセット直接生成不可
- Claude は ScriptableObject の `.asset` ファイルを直接生成できない（Unity の GUID 計算・SerializedObject 経由のシリアライズが必要なため）。
- SO アセットが必要な場合は **Editor 拡張ツールを作成 → Unity Editor のメニュー経由でユーザーに実行依頼** する流れ。詳細手順は [920 §2.3](920_implementation_notes.md)。

### 7.9 コード内コメントの最小スコープ
- **コメントの役割は「コメントだけで実装内容が伝わる」こと**。
- **書かないもの**：
  - `Docs/` 参照（「仕様詳細は 320 §3.2 参照」「100 §4.1 より」等）→参照するのではなく、必要な実装内容だけをコメントに書き直す
  - 経緯・背景・作業ログ（「Phase X で追加」「Y の経緯で〜」「旧仕様では〜」等）
  - 未来計画・予定（「将来〜する予定」「バランス調整フェーズで判断」「Step X で対応」等）
  - コミットハッシュ・Issue 番号
  - パス転写（`// Assets/Scripts/...`）
- **理由**：実装者が気にすべきは「現在の正確な仕様」だけ。経緯・未来計画・外部参照は git log と仕様書側で十分で、コードに書くと読み手のノイズになる上に保守できない。

### 7.10 未使用の枠を作らない（死蔵防止）

- **enum 値／プロパティ／フィールド／メソッドを「将来用」を理由に追加しない**。追加するなら同コミット内で読む側（適用機構）もセットで書く。
- 「拡張点を先回りで担保しよう」という設計癖が悪く出るパターン。読む側がないまま放置されると **「動くと思って選んだら何も起こらない」型のバグ** を生む。発見コストはバグ調査経由になり高い。
- 過去事例：`UpgradeKind.EvasionBoost` / `WazaPowerBoost` は「enum 値と TargetWazaId フィールドだけ用意・読む側ゼロ」で死蔵し、ユーザー試遊で「EVA +5 Upgrade を選んでも効かない」と報告されて初めて発覚（2026-06-19）。Lv 強化機構の根本再整理コストを誘発した。
- 「とりあえず enum だけ足しておこう」は禁止。少なくとも 1 つの実 case を読む側に書くのが最小単位。

### 7.11 同パターン 3 個目で構造化を検討

- if / switch の case を増やすとき、**3 個目を書く前に「データ駆動／パターン抽出に切り出すべきか」を自問する**。1〜2 個は許容、3 個目で構造化検討を提案する。
- 「とりあえず if を 1 個足す」を繰り返すと、新規 case 追加時に 3〜5 箇所同期メンテが必要な保守不能コードになる（OCP 違反）。
- 過去事例：`FormatKindShort` / `FormatKindLong` / `GetMagnitude` / `EffectFactory.CreateByKind` が EffectKind ごとに散らばっており、新規 EffectKind 追加時に 3〜5 箇所同期メンテが必要（[912 §設計上の弱点メモ](912_vsprototype_devlog_2.md)）。
- 「最終判断はユーザー」だが、Claude 側は **自問した上で構造化案を提示** する責務を負う（ユーザーがコード中身を見ない前提で、Claude が代わりに目を光らせる）。

### 7.12 節目レビューを自発的に提案

- 大きな機能完了タイミング（コミット 3〜5 個まとめてクローズ時など）で、**Claude から「ついでに周辺の死蔵／条件分岐肥大／その場しのぎを点検しますか？」と提案** する。
- ユーザーがコード中身を見られない前提なので、Claude が代わりに目を光らせる役。`/code-review` スキルや簡易 grep 点検で機械的に拾えるものから着手。

## 8. ドキュメント執筆規約

仕様書（`100_*` / `110_*` / `300_*` / `310_*` 等）・設計書（`500_*`）すべてに適用する。

### 8.1 設計書は「責務・抽象・判断」だけを書く

- **書く**：層構成・責務分割・抽象 interface 定義・設計判断（Why）・仕様要素 → 実装観点マッピング
- **書かない**：プロパティ列挙・メソッドシグネチャ・数式の詳細・処理フロー詳細
- **理由**：詳細はコードと仕様書（100/110）に存在する。設計書は読み物として薄く保つ方が更新され・参照される

### 8.2 「仕様要素 → 実装観点」マッピング表を骨格にする

仕様書の要素（例：「ラウンド開始時オートセーブ」）と、設計上の実装観点（例：「`IEventBus` + `ISaveStore`」）を表で対応付ける。テキスト段落で説明するより、表で対応関係を一目で示す方が価値が高い。

仕様書側（100/110/300/310 等）でも、決定事項は表で示すことを優先する。

### 8.3 経緯・作業ログ・過去のバグ事例は書かない

- **書かない**：「YYYY-MM-DD に再設計」「過去のバグでこうなった」「Phase 0-X で実装した」等の経緯・実装ログ
- **書く**：現在の決定事項のみ。「なぜそうなっているか」の Why は決定事項として残してよい（ただし「過去にこうだったから」ではなく「こういう原則に従う」の形式で）
- **理由**：経緯は git log / コミットメッセージ / メモリで追えるが、設計書に残すと読み手のノイズになる

### 8.4 二重ドリフト防止 — 詳細は 1 箇所だけ

同じ情報（プロパティ名・数式・数値）を仕様書とコード両方に書くと、片方が更新されたとき必ず腐る。

- **コードで自明な情報はコードに任せる**：クラス名・プロパティ列挙・interface のメソッド一覧
- **仕様書にしかない情報は仕様書に**：数式・数値・処理フロー
- **設計書はどちらも繰り返さない**：「どこに何があるか」の参照リンクだけにする

### 8.5 退避ルール（既存 §4.2 と整合）

陳腐化したドキュメントは削除せず `Docs/_old/` に退避する。設計書を全面再設計したときも、旧版は `_old/` に移動して新版の冒頭から参照リンクを張る。

### 8.6 ドキュメントからメモリへの参照禁止（一方向参照）

- **ドキュメント側にメモリリンク（`[[memory-name]]` 形式）を書いてはいけない**。ドキュメントは人間が読むものであり、メモリは人間からは追えない＝リンクが死ぬ＋情報が分散して全体像が掴めなくなる。
- 参照の方向は**メモリ → ドキュメント**の一方向に統一する：
  - **メモリ側**：ドキュメントへのリンク（`[Docs/XXX.md](path)` 形式）を書いてよい。
  - **ドキュメント側**：他ドキュメントへのリンクのみ書く。メモリ参照は書かない。
- 既存のメモリリンクを発見したら、ドキュメント内で完結する文言に置換する（理由を 1 行で展開・もしくは削除）。

### 8.7 910 番台ドキュメントの位置付け

`910_*` は **Claude 用の作業ログ・経緯メモ**。仕様の正ではない。

- 仕様書（100/110/300/310/330/400/500）から 910 へは原則リンクしない。910 → 仕様書のリンクは自由
- §8.1〜§8.5 の「責務・抽象・判断だけ書く／経緯ログ禁止／二重ドリフト防止」ルールは**適用外**。経緯・作業ログ・コミットハッシュを思いつくまま追記してよい
- 一括整合性修正の対象外（仕様書セクション番号振り直し・ID 系統一斉切替等は触らない）
